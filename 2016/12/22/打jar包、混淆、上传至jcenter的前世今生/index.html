Android 打jar包、混淆、上传至jcenter的前世今生

Posted on 2016-12-22   |     |   visitors

我们做能力提供的时候，SDK的提供方式一般为jar,或者是aar，而且提供jar的时候需要对其进行混淆。现在Android项目都用gradle进行构建,所以摆在我们面前的问题就是：利用gradle 打jar包、混淆jar包、上传jar包至jcenter。

1.gradle 打jar包

gradle打jar包、以及怎样将dependencies里面的第三方library打入jar包里面，这个问题已经在《Android studio build jar with dependence》里面进行详细地阐述了。

2.gradle 混淆jar包

sdk的混淆有很多限制，对外的接口都不能混淆、对外的实体类都不能混淆。先贴出混淆代码：

task buildJar(type: Jar) {
    appendix = "demo"
    baseName = "androidJar"
    version = "1.0.0"
    classifier = "release"
//后缀名
    extension = "jar"
//最终的 Jar 包名，如果没设置，默认为 [baseName]-[appendix]-[version]-[classifier].[extension]
    archiveName = "sdk.jar"

//需打包的资源所在的路径集
    def srcClassDir = [project.buildDir.absolutePath + "/intermediates/classes/release/"];
//初始化资源路径集
    from srcClassDir
//去除路径集下部分的资源 from srcClassDir
//    exclude "org/chaos/demo/jar/MainActivity.class"
//    exclude "org/chaos/demo/jar/MainActivity/$*.class"
    //去掉不需要打包的目录和文件
    exclude('test/', 'BuildConfig.class', 'R.class')
//去掉R$开头的文件
//    exclude { it.name.startsWith('R$'); }

//去掉不需要打包的目录和文件
    exclude('test/',  'androidTest/')
//    exclude('META-INF/')
//去掉R$开头的文件
    exclude {
        it.name.startsWith('R$');
        it.name.startsWith('org.apache.thrift');
        it.name.startsWith('META-INF');
    }
    //注: exclude include 支持可变长参数
    exclude "im/"
}

task createJar(type: Jar) {
baseName = "littlec"
from {
    List list = (configurations.compile - configurations.retain).
            collect {
                it.isDirectory() ? it : zipTree(it).matching {
                   exclude { it.path.contains('META-INF') }
                }
            }
    List<Object> removeList = new ArrayList<>();
    for (Object object : list) {
        println(object.toString());
        if (object.toString().contains("jsr")) {
            removeList.add(object)
            println("jsrObject remove")
        } else if (object.toString().contains("javax.annotation")) {
            removeList.add(object)
            println("annotation remove")
        } else if (object.toString().contains("MiPush_SDK_Client")) {
            removeList.add(object)
            println("MiPush_SDK_Client remove")
        }
    }
    list.removeAll(removeList)
    return list;
}
from fileTree(dir:'src/main',includes:['assets/**'])

}

task confuseJar(type: proguard.gradle.ProGuardTask, dependsOn: ['createJar']) {

// 未混淆的jar路径
injars 'build/libs/sdk-'+version+'.jar'
// 混淆后的jar输出路径
outjars 'build/libs/sdk-'+version+'-confuse.jar'
// 混淆协议
configuration 'proguard-rules.pro'

}

上面的confuseJar就是混淆的task，该task是基于createJar的，执行完会在/build/libs/下面产生一个sdk.jar包，confuse指定该jar包为未混淆的jar包导入，并且指定混淆后的jar包输出路径，指定混淆的规则文件，这个就是工程下面的pro文件，下面的pro文件的内容：

在pro里面，需要保护对外暴露的实体类、还有实现在类里面的广播内部类，不然会报onReceive没有实现这样的错误。上面的pro文件都添加了注释。

3.上传至Jcenter

对于我们做能力开发的，如果不能做到远程依赖，每次发布版本，客户都需要下一次jar包进行替换，这个特别麻烦。为了解决这个问题，看了很多网上的资料，踩了很多坑，在这里梳理一下这些踩过的坑。

3.1登录与注册bintray

打开bintray,选择使用google或者github登录即可：

3.2创建组织

点击右上角的头像，选择create Organization->create new organization->create,命名的时候需要慎重，因为后面需要在下面新建项目的。

3.2创建仓库

创建完组织后，需要在下面创建仓库，这个仓库用来放你上传的代码，仓库类型和仓库名称都要跟后面介绍的上传任务里面的名称一致。
点击Add New Repository：

3.3创建项目

创建仓库后，在仓库下面创建项目，项目名称用sdk名称。

##3.4 使用maven插件和bintray插件 ##
在根目录build.gradle下面加入mavencentral()以及maven 和 bintray插件的路径
代码如下：

buildscript {
    repositories {
        jcenter()
        mavenCentral()

    }
    dependencies {
        classpath 'com.android.tools.build:gradle:2.2.3'
        classpath 'com.jfrog.bintray.gradle:gradle-bintray-plugin:1.0'
        classpath 'com.github.dcendents:android-maven-gradle-plugin:1.5'
        classpath "com.google.protobuf:protobuf-gradle-plugin:0.8.0"
    }
}

在对应的子工程build.gradle下面加入二种插件的使用,在apply plugin:’com.android.library’下面加入如下代码：

apply plugin: 'com.github.dcendents.android-maven'
apply plugin: 'com.jfrog.bintray'

3.5 加入bintray任务
首先在根目录下面的local.properties 下面添加bintray username和apikey
username就是进入bintray里面右上角图像边上显示的名称。
apikey可以在Edit Your Profile里面找到里面找到。
特别需要指出的是组织名称需要指定，不然会默认为上面的username,这样bintrayUpload肯定会失败。
上传部分代码如下：

def siteUrl = '******'    //Homepage URL of the library

}

上面archives file那里指定的是上面混淆过的jar包，这是根据我们自己的需求来的，因为在线依赖下来的工程项目在External Libraries里面都能看到，但是我们sdk的代码有些部分可能比较敏感，还是想稍微混淆一下，我就考虑了这个流程： 先打jar包，然后将该jar包混淆，最后将混淆后的jar包上传至jcenter。

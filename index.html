git常用操作

Veröffentlicht am 2016-12-27   |     |   visitors

git虽然命令相当多，但是记住下面这些命令就够用了。这些都是使用频率很高的命令：

git init 初始git库

git add 添加文件到缓存区

git commit -m 提交到仓库

git status 查看当前的状态

git log 可以查看提交的记录

git reset -hard id 版本会退到某一个commit_id
commitid可以从git log查看

git diff HEAD – filename可以比较工作区和暂存区有啥的版本有啥区别

git checkout – filename或者git reset HEAD file 撤销修改操作

git rm filename 从版本库里面删除某个文件

git checkout -b dev 创建新分支

git branch 查看当前分支

git checkout -b dev新建并且切换分支到dev

git checkout branchname 切换分支

git branch -d 删除分支

git branch 创建分支

git merge 合并分支

git stash 暂时保存现场，这样工作区就是干净的，master也是 干净的

git stash apply 恢复现场

git stash pop 删除现场

git push remote-name

git checkout -b test origin/dev 将远程仓库dev分支拉取到本地,并且新建test分支。建立本地分支和git branch –set-upstream
branch-name origin/branch-name建立本地分支与远程的关联

git tag 打一个新标签

git tag 查看所有标签

git show 查看标签

git push original –tags 一次性推送全部尚未推送到远程的本地标签

git tag -d 删除本地标签

git push origin :refs/tags/ 可以删除一个远程标签

Android 打jar包、混淆、上传至jcenter的前世今生

Veröffentlicht am 2016-12-22   |     |   visitors

我们做能力提供的时候，SDK的提供方式一般为jar,或者是aar，而且提供jar的时候需要对其进行混淆。现在Android项目都用gradle进行构建,所以摆在我们面前的问题就是：利用gradle 打jar包、混淆jar包、上传jar包至jcenter。

1.gradle 打jar包

gradle打jar包、以及怎样将dependencies里面的第三方library打入jar包里面，这个问题已经在《Android studio build jar with dependence》里面进行详细地阐述了。

2.gradle 混淆jar包

sdk的混淆有很多限制，对外的接口都不能混淆、对外的实体类都不能混淆。先贴出混淆代码：

task buildJar(type: Jar) {
    appendix = "demo"
    baseName = "androidJar"
    version = "1.0.0"
    classifier = "release"
//后缀名
    extension = "jar"

    def srcClassDir = [project.buildDir.absolutePath + "/intermediates/classes/release/"];
//初始化资源路径集
    from srcClassDir
//去除路径集下部分的资源 from srcClassDir
//    exclude "org/chaos/demo/jar/MainActivity.class"
//    exclude "org/chaos/demo/jar/MainActivity/$*.class"
    //去掉不需要打包的目录和文件
    exclude('test/', 'BuildConfig.class', 'R.class')
//去掉R$开头的文件
//    exclude { it.name.startsWith('R$'); }

//去掉不需要打包的目录和文件
    exclude('test/',  'androidTest/')
//    exclude('META-INF/')

    exclude {
        it.name.startsWith('R$');

        it.name.startsWith('META-INF');
    }
    //注: exclude include 支持可变长参数
    exclude "im/"
}

task createJar(type: Jar) {
baseName = "littlec"
from {
    List list = (configurations.compile - configurations.retain).
            collect {
                it.isDirectory() ? it : zipTree(it).matching {
                   exclude { it.path.contains('META-INF') }
                }
            }
    List<Object> removeList = new ArrayList<>();
    for (Object object : list) {
        println(object.toString());
        if (object.toString().contains("jsr")) {
            removeList.add(object)
            println("jsrObject remove")
        } else if (object.toString().contains("javax.annotation")) {
            removeList.add(object)
            println("annotation remove")
        } else if (object.toString().contains("MiPush_SDK_Client")) {
            removeList.add(object)
            println("MiPush_SDK_Client remove")
        }
    }
    list.removeAll(removeList)
    return list;
}

task confuseJar(type: proguard.gradle.ProGuardTask, dependsOn: ['createJar']) {

// 未混淆的jar路径
injars 'build/libs/sdk-'+version+'.jar'

}

在pro里面，需要保护对外暴露的实体类、还有实现在类里面的广播内部类，不然会报onReceive没有实现这样的错误。上面的pro文件都添加了注释。

3.上传至Jcenter

对于我们做能力开发的，如果不能做到远程依赖，每次发布版本，客户都需要下一次jar包进行替换，这个特别麻烦。为了解决这个问题，看了很多网上的资料，踩了很多坑，在这里梳理一下这些踩过的坑。

3.1登录与注册bintray

打开bintray,选择使用google或者github登录即可：

3.2创建组织

点击右上角的头像，选择create Organization->create new organization->create,命名的时候需要慎重，因为后面需要在下面新建项目的。

3.2创建仓库

创建完组织后，需要在下面创建仓库，这个仓库用来放你上传的代码，仓库类型和仓库名称都要跟后面介绍的上传任务里面的名称一致。
点击Add New Repository：

3.3创建项目

创建仓库后，在仓库下面创建项目，项目名称用sdk名称。

##3.4 使用maven插件和bintray插件 ##
在根目录build.gradle下面加入mavencentral()以及maven 和 bintray插件的路径
代码如下：

buildscript {
    repositories {
        jcenter()
        mavenCentral()

    }
    dependencies {
        classpath 'com.android.tools.build:gradle:2.2.3'
        classpath 'com.jfrog.bintray.gradle:gradle-bintray-plugin:1.0'
        classpath 'com.github.dcendents:android-maven-gradle-plugin:1.5'
        classpath "com.google.protobuf:protobuf-gradle-plugin:0.8.0"
    }
}

在对应的子工程build.gradle下面加入二种插件的使用,在apply plugin:’com.android.library’下面加入如下代码：

apply plugin: 'com.github.dcendents.android-maven'
apply plugin: 'com.jfrog.bintray'

3.5 加入bintray任务
首先在根目录下面的local.properties 下面添加bintray username和apikey
username就是进入bintray里面右上角图像边上显示的名称。
apikey可以在Edit Your Profile里面找到里面找到。
特别需要指出的是组织名称需要指定，不然会默认为上面的username,这样bintrayUpload肯定会失败。

一 EditTextView

Emojicon对于常用的TextView和EditText做了自定义控件。自定义控件大致的步骤如下：

1.我们的自定义控件和其他的控件一样,应该写成一个类,而这个类的属性是是有自己来决定的.

2.我们要在res/values目录下建立一个attrs.xml的文件,并在此文件中增加对控件的属性的定义.

3.使用AttributeSet来完成控件类的构造函数,并在构造函数中将自定义控件类中变量与attrs.xml中的属性连接起来.

4.在自定义控件类中使用这些已经连接的属性变量.

5.将自定义的控件类定义到布局用的xml文件中去.

6.在界面中生成此自定义控件类对象,并加以使用.

自定义属性 Emojicon：

表情要能正确地在TextView里面显示，必然要在setText里面做转换操作：

     public void setText(CharSequence text, BufferType type) {
         if (!TextUtils.isEmpty(text)) {
        SpannableStringBuilder builder = new SpannableStringBuilder(text);
        EmojiconHandler.addEmojis(getContext(), builder, mEmojiconSize, mEmojiconAlignment, mEmojiconTextSize, mTextStart, mTextLength, mUseSystemDefault);

这里将输入的文本，用EmojiconHandler.addEmojis将文本转换成spannableString,就是将文本里面的表情unicode用本地表情图片替换，再在TextView里面显示。SpannableString、SpannableStringBuilder基本上与String差不多，也是用来存储字符串，但它们俩的特殊就在于有一个SetSpan（）函数，能给这些存储的String添加各种格式或者称样式（Span），将原来的String以不同的样式显示出来，比如在原来String上加下划线、加背景色、改变字体颜色、用图片把指定的文字给替换掉，等等。所以，总而言之，SpannableString、SpannableStringBuilder与String一样， 首先也是传字符串，但SpannableString、SpannableStringBuilder可以对这些字符串添加额外的样式信息，但String则不行。

我们项目组专门做能力开发的，前面的版本都是用eclipse开发的，后面迁移到android studio还是用eclipse里面的FatJar打包。这个问题拖了一个月，最近突击了一下，解决了这个问题。希望能够让更多人吸取经验。
对gradle不熟悉的可以看看 Gradle User Guide中文版https://dongchuan.gitbooks.io/gradle-user-guide-/content/tutorials/

一. 分析dependencies

指令gradle App:dependencies 可以列出所有的依赖树，通过依赖树可以分析出自己的Jar包需要包括哪些第三方jar。

二. 打包src下面的java文件

首先先来看看怎样将自己src文件夹下面的代码打包，不废话，直接贴出代码：

task buildJar(dependsOn: [build], type: Jar) {
appendix = "demo"
baseName = "androidJar"
version = "1.0.0"
classifier = "release"
//后缀名
extension = "jar"
//最终的 Jar 包名，如果没设置，默认为 [baseName]-[appendix]-[version]-[classifier].[extension]
archiveName = "littlec.jar"

//需打包的资源所在的路径集
def srcClassDir = [project.buildDir.absolutePath + "/intermediates/classes/release/"];
//初始化资源路径集
from srcClassDir

//去除路径集下部分的资源
//exclude "org/chaos/demo/jar/MainActivity.class"
//exclude "org/chaos/demo/jar/MainActivity/$*.class"
//去掉不需要打包的目录和文件
exclude('test/', 'BuildConfig.class', 'R.class')
//去掉R$开头的文件
exclude { it.name.startsWith('R$'); }

//去掉不需要打包的目录和文件
exclude('test/', 'BuildConfig.class', 'R.class', 'androidTest/')
exclude('META-INF/')
//去掉R$开头的文件
exclude { it.name.startsWith('R$');
//it.name.startsWith('org.apache.thrift');
it.name.startsWith('META-INF');
}
//注: exclude include 支持可变长参数
 exclude "im/"
}

这个task 里面可以将src里面的代码打包成指定的名称如littlec.jar，生成目录在/build/libs/下面。
注意点：
打包之前需要确认/INTERMEDIATES/CLASSES/RELEASE/下面存在文件，如果不存在则需要重新编译一下，重新编译BUILD->MAKE PROJECT

三. 将依赖远程仓库和本地依赖打包

Android工程里面有很多在线依赖以及项目工程里面libs下面的本地依赖，如果想将这些依赖和自己的代码打成一个Jar包，我们应该怎么办？这个点让我费解了好久，其实我们可以发现：依赖远程仓库的第三方包也会缓存到本地，那么我们怎么用gradle找到这些第三方依赖？后面查资料发现configurations.compile.collect就可以遍历依赖里面的jar包，我们只需要再看一下每个元素是不是还有子树，而zipTree可以遍历一个文件下面的子文件，所以上面Task 的from 可以改为

configurations.compile.
 collect {
it.isDirectory() ? it : zipTree(it)
}

输入明确了，只需要将每个依赖里面的META_INF目录去掉就行了。那就稍微改进一下：

 configurations.compile.
 collect {
it.isDirectory() ? it : zipTree(it).matching {
exclude{it.path.contains('META-INF')}
}
}

如果是文件夹对其文件路径都匹配一下META-INF，进而排除 META-INF文件夹。
上面排除了META-INF等不需要的文件，有时候我们不想将所有在dependencies里面依赖的第三方包打包，那么我们怎么很优雅地将这些包进行删选了？而且不影响正常的编译。只需要在代码里面加入

configurations{
retain
compile.extendsFrom retain
}

并且将dependencies里面将自己不想打包的compile 修改为 retain。如下所示：

dependencies {

retain 'javax.annotation:javax.annotation-api:1.2'

}

最后将上面from里面的代码修改如下：

`(configurations.compile-configurations.retain).
       collect {
        it.isDirectory() ? it : zipTree(it).matching {
            exclude{it.path.contains('META-INF')}
        }
    }`

这样javax.annotation:javax.annotation-api:1.2 这个包就不会被打进最终的Jar包。
最终的task代码如下，这个task 基于上述的buildJar，这样就可以将src下面的代码和第三方依赖的代码打成一个jar包。

一 字符集和字符编码

字符集就规定了某个文字对应的二进制数字存放方式（编码）和某串二进制数值代表了哪个文字（解码）的转换关系。比如常见的UTF-8、UTF-16、GBK等。
字符编码简单来说就是一种编码字符集和实际存储数值之间的转换关系。有三个关键元素：字库表、编码字符集、字符编码。
理解了这二个概念就能理解UTF-8和Unicode、UTF-16和Unicode之间的关系，Unicode就是编码字符集、UTF-8、UTF-16是字符编码。
详细知识请参考http://cenalulu.github.io/linux/character-encoding/

二 标准表情库

Emoji表情最初从小日本的“绘文字”演化过来，刚开始流传最广的是日本软银的Softbank编码了，在ios初期也是采用这种编码。2013年，谷歌终于推出了自己的开源color font。但是android的表情很不美观，而且低版本不支持部分表情，这对于某些IM应用来讲是无法忍受的，这就需要我们实现一个表情库，实现表情的兼容和美观。常规表情以及unicode的对应表可参考http://apps.timwhitlock.info/emoji/tables/unicode

   private static String convertUnicode(String emo) {
     emo = emo.substring(emo.indexOf("_") + 1);
     if (emo.length() < 6) {
             String s = new String(Character.toChars(Integer.parseInt(emo, 16)));
         return s;
     }
     String[] emos = emo.split("_");
     char[] char0 = Character.toChars(Integer.parseInt(emos[0], 16));
     char[] char1 = Character.toChars(Integer.parseInt(emos[1], 16));
     char[] emoji = new char[char0.length + char1.length];
     for (int i = 0; i < char0.length; i++) {
         emoji[i] = char0[i];
     }
     for (int i = char0.length; i < emoji.length; i++) {
         emoji[i] = char1[i - char0.length];
     }
     String s =  new String(emoji);
     return s;
